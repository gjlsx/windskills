# Agent Rules

Follow `.agent-rules.md` before any action.

## Core Principles

- Read rules first, then read the tasklist, then execute.
- Use markdown files as the primary source of truth.
- Important process decisions must be documented.
- Skills are passive documentation under `skills/`.
- Executor can hold only one active lock at a time.
- Review is self-review unless local rules say otherwise.
- Memory is split into global memory and project-specific memory.
- all task done must had commit ,and update tasklist fields with the commit hash. 

## Tasklist Requirement

- Before building a tasklist, read `tasklist_rules.md`.
- Before running a task from a tasklist, read `tasklist_rules.md`.
- to backup a done tasklist ,move it to /docs/backuptask/

## Allowed Files To Modify

An executor may modify only:

1. its own task log under `tasks/<agent>/`
2. its own lock under `tasks/locks/<agent>_<taskid>.lock`
3. allowed status fields in the active tasklist file， and there is 
  no other `ainame.taskid.lock` file in `xx.lock` files dir(like:`/tasks/locks/`)   
4. code, tests, docs, or config directly required by the claimed task

## 1. Allowed Files to Modify or start a task  only if 1,2,3,4 below:
  1- its own task log under `tasks/<agent>/`
  2 - its own lock under `tasks/locks/ainame_taskid.lock` ,
  3 - and there is no other `ainame.taskid.lock` file in /tasks/locks/   
   means if there is file `B.taskid.lock` in the dir ,Then ai A can't do this task,
  4 - allowed status fields in `tasklistall.md` or input/linkto `taskxxx.md` file!


Do not start a task if another agent already holds the task lock.

## Forbidden Actions

Executor must not:

- create new tasks unless explicitly asked to build a tasklist
- change the tasklist table structure
- modify other agent logs
- modify other agent locks
- modify unrelated memory files
- mark a task `done` without passing verification

## Startup Sequence

1. Read required rule and memory files. Read `.agent-rules.md` , `tasklist_rules.md`,Read target tasklist
2.3.4 steps skip because it is in step 1.
5. scan `tasks/locks/` or project's lock files dir for your own lock using filename pattern: `<agent name>_<taskid>.lock`
6. if your own lock exists:
   - inspect related task status in `tasklistall.md`  or input/linkto `tasklistxxx.md` file!
   - if task is already `done` or `cancelled`, delete the lock
   - otherwise resume that task directly
7. if no active own lock exists, find first matching `todo` task for your role
8. check dependencies
9. create lock
10. claim task
11. execute
12. run tests
13. self code review
14. if refactor is needed, repeat execute -> test -> self review until no refactor is needed
15. After done a task, the executor must commit all changed files with  message describing what was done (if it is task then msg must include whole-taskid like : `codex_t03160314.p10`).
16. update tasklist ,if had commit you must update hash to the tasklist 
17. remove lock 

## Locked Task Rule

- Lock is never taken over by another AI.
- Each AI scans only its own lock first.
- One AI can have only one active lock at a time.
- If an AI already has an unfinished lock task, it must continue that task first.
- Lock filename format looks like : '(ai's name)_taskid.lock` ,
  - `codex_t03160314.p10.lock`  
  - `claude_t03160314.p11.lock`

## Claim Rule

When claiming a task, update only allowed tasklist fields:

- `Owner`
- `Status`
- `Claim`
- `Finish`
- `Report`
- `Git`
- `Review`
- `Score`

Claim format:

`startat:YYMMDDHHMMSS <agent> tasks/<agent>/<taskid>.md`

## Execution Loop

If the task is already claimed by the current AI, do not restart the full claiming flow. Go directly to:

1. execute
2. verify
3. self-review
4. repeat if refactor is needed
5. commit all changed files with  msg describing what was done (if it is task then msg must include whole-taskid like : `codex_t03160314.p10`).  When committing a bug fix (fix), the commit msg MUST include an ISSUE-ID/taskid(like: FIX: `t260321.p945`).  with [P0/P1/P2] and andy detail info, -P2(Can be delayed,Non-critical) P1 (Important Should be fixed now) -P0 (Blocking,crash... must be fixed immediately)
6. update tasklist (include commit hash)
7. remove lock

## Testing Rule

A task cannot become `done` unless required verification passes.

Examples:

- `pytest`
- `npm test`
- `go test ./...`
- project-specific manual validation with evidence

If no tests exist, add minimal validation or document manual verification evidence.

## Self Review Rule

Required checks:

- requirement satisfied
- verification passed
- no obvious duplicate logic left unhandled
- naming consistent
- no required refactor remains
- task log is complete

Review outcomes:

- `pass:no-refactor-needed`
- `pass:minor-refactor-done`
- `partial:needs-followup`

## Definition Of Done

A task is `done` only when all are true:

- implementation complete
- required tests passed
- self review says no further necessary refactor
- task log written
- commit is ok
- tasklist updated
- lock removed

## Task Status Set

Allowed statuses:

- `todo`
- `doing`
- `blocked`
- `partial`
- `pending`
- `done`
- `cancelled`

## Dependency Rule

Before claiming a `todo` task, check `Depends`. If any dependency is not `done`, skip the task.

## Memory Read Rule

Memory root is `memorys/`.

Use:

- `memorys/global.md` for global rules and project memory links
- `memorys/<project-name>/memory.md` for project-specific memory

Do not read unrelated project memory unless explicitly required.
