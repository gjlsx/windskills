# Agent Rules

Follow `.agent-rules.md` before any action.

## Mandatory Commit Chain

Every task status change that updates the tasklist row must be committed.

Required commit checkpoints:

1. After changing a task row from `todo` to `doing`, commit the tasklist file.
2. After task implementation is complete, commit all implementation/test/doc/config files required by the task.
3. Record the real task commit hash in `tasks/tasklog/<agent>_<taskid>.md`.
4. Write the same hash plus terminal task fields back to the tasklist row.
5. Commit the tasklist state update. This commit must include the tasklist file itself, and may also include the task log if the log changed after the task commit.
6. After moving a tasklist into backup, make a separate archive commit.

Status-change commits are mandatory for:

- `todo -> doing`
- `doing -> done`,or `blocked`,`partial`,`cancelled`
- tasklist archive

## Core Principles

- Read rules first, then read the tasklist, then execute.
- Use markdown files as the primary source of truth.
- Important process decisions must be documented.
- Skills are passive documentation under `skills/`.
- Executor can hold only one active lock at a time.
- Review is self-review unless local rules say otherwise.
- `tasklist_rules.md` is the single source of truth for required tasklist structure, fixed columns, task status vocabulary, and `TaskID` format.
- `tasklistall.md` is only a tasklist index. It is not a task execution board.
- Task detail docs describe task requirements and long-form acceptance notes.
- Task logs describe actual execution, verification, and finish status.
- When a task uses a detail doc, the task log should reference that same `Detail` path.

## Default Paths

Unless this repository explicitly changes them, use these default paths:

- tasklist index: `tasklistall.md`
- global memory: `memorys/global.md`
- lock dir: `tasks/locks/`
- task log dir: `tasks/tasklog/`
- task detail dir: `tasks/details/`
- default concrete tasklist dirs:
  - repository root: `./`
  - backup dir: `docs/backuptask/`

`tasklistall.md` is an index file by default. Resolve a concrete `tasklistMMDDhhmm.md` from it before executing tasks.

Default resolution rules:

1. If the user explicitly gives a tasklist path, use it directly.
2. Otherwise read `tasklistall.md` and resolve an `active` entry using `Dir + Name`.
3. If more than one `active` entry exists, do not guess. Require user selection or tasklist-creator instruction.
4. If the indexed path is missing, check `./` and `docs/backuptask/`.
5. If still missing, search the configured concrete tasklist dirs and then the repository for the indexed filename.
6. If no concrete tasklist exists anywhere on first run, create a default `tasklistMMDDhhmm.md` in `./` and append it to `tasklistall.md`.

Default `tasklistall.md` format:

| Name | Time | User | Status | Dir | Desc |
|---|---|---|---|---|---|
| tasklist03170443.md | 2603170443 | wind | active | ./ | Likeshop 一期遗留收口任务 0317（taskexec） |
| tasklist03172344.md | 2603172344 | wind | active | ./ | Likeshop 用户优先级 P0 闭环 E2E 验收 0317（taskexec） |

If files/directories are missing on the first run, bootstrap them using these default paths and record the active paths in this `.agent-rules.md`.

## Tasklist State

Tasklist-level states are:

- `active`: the tasklist is still live and may receive new follow-up rows
- `archived`: the tasklist has been moved to backup and no longer accepts new rows

`done` belongs only to task rows. It is not a tasklist-level status in `tasklistall.md`.

Any task row that is already `done` must never have its status changed again. Follow-up work must be tracked by appending a new row.

## Follow-up Bugfix Rule

Default policy:

- Reuse the same active, non-archived tasklist when the bugfix belongs to the same workflow or scope.
- If the original task row is already `done`, add a new task row instead of rewriting the completed row.
- Only the tasklist creator may append follow-up rows or create/update `tasks/details/d_<taskid>.md`.
- If the creator is unavailable, stop and hand the decision back to a human.
- Follow-up rows must set `Type` to `bugfix`.
- Include `FIX:<source_taskid>` in the task `Description` or the referenced `Detail` file.
- Same-workflow follow-up bugfix rows are part of the taskexec workflow. They do not require an `ISSUE-ID` in the task row or task commit message unless a repository explicitly wants both.
- If the tasklist is already `archived`, or the bugfix crosses scope, milestone, or version boundaries, create a new tasklist instead.

## Archive Sweep Rule

Archive sweeps are allowed only in configured concrete tasklist dirs, including the repository root tasklist dir and the configured backup dir.

Run an archive sweep in these cases:

- before creating a new tasklist
- immediately after registering a tasklist in `tasklistall.md`
- when the user explicitly asks to archive/submit
- immediately after archiving any tasklist

Archive sweep behavior:

1. Read `tasklistall.md`.
2. Resolve every indexed `active` tasklist using its `Dir + Name`, then verify against the repository root tasklist dir and any configured concrete tasklist dirs.
3. For each resolved active tasklist, if every task row is `done`, update the tasklist status to `archived`.
4. Move that tasklist file into the configured backup dir.
5. Update the matching `tasklistall.md` row from `active` to `archived`.
6. Commit the archive change, including `tasklistall.md`.

Never include template/example files outside the configured concrete tasklist dirs in an archive sweep.


## Rule: Direct Fix Commits Outside Taskexec MUST Include Issue ID
  This rule applies only to direct fix commits that do not use the taskexec workflow.
  When committing a direct fix outside taskexec, the commit message MUST include an ISSUE-ID with [P0/P1/P2].
  ISSUE-ID get from `/scripts/new_issue.sh`
  Canonical issue docs live under `docs/issues/` with one issue per file:
      `docs/issues/ISSUE-XXX.md`
  `docs/issue_index.md` is the summary index only and must use 5 columns:
      `IssueID | Priority | Status | Summary | detailFile`
  The last column `detailFile` must be the real repo-relative path of the issue detail file.
  ✅ Correct format:
      fix(scope): resolve ISSUE-XXX [P1] + any detail info just like:
      ((Recommended))
      bug: xxxx
      changes: xxx
      verify:xxx
      docs:xxx
      ....and so on..
  ❌ Invalid:
     fix: login bug ok
  -P2(Can be delayed,Non-critical) P1 (Important Should be fixed now)
  -P0 (Blocking,crash... must be fixed immediately)
  Taskexec task commits follow the task commit rules below and are not forced to include an `ISSUE-ID`.
All direct fix commits outside taskexec MUST include ISSUE-ID and Priority.

## Allowed Files To Modify

An executor may modify only the following files during normal execution:

1. the task log under `tasks/tasklog/<agent>_<taskid>.md`
2. its own lock under `tasks/locks/<agent>_<taskid>.lock`
3. executor-editable fields in the active tasklist file, provided there is no other `<agent>_<taskid>.lock` file in the lock dir
4. code, tests, docs, or config directly required by the claimed task

Task definition files are separate:

- `tasks/details/d_<taskid>.md` is read-only during normal execution
- only the tasklist creator may create/update task detail docs while building or revising task definitions

Do not start a task if another agent already holds the task lock.

## Forbidden Actions

Executor must not:

- create new tasks unless explicitly asked to build a tasklist or to append a creator-approved follow-up task
- change the tasklist table structure
- modify other agent logs
- modify other agent locks
- modify unrelated memory files
- mark a task `done` without passing verification
- rewrite a completed row into a follow-up bugfix row
- append a follow-up row when the tasklist creator has not approved it

## Startup Sequence

1. Read required rule and memory files. Read `.agent-rules.md`, `tasklist_rules.md`, and the target tasklist file.
2. If no target tasklist file was provided, read `tasklistall.md` as the index and resolve the target concrete `tasklistMMDDhhmm.md` file using the default resolution rules in this file.
3. Read the selected task row's `Detail` file when the `Detail` column is not empty.
4. Read `memorys/global.md` if it exists or use the path defined in this file.
5. Use the default paths only when this file does not define another path.
6. Scan `tasks/locks/` or the project's lock dir for your own lock using filename pattern `<agent>_<taskid>.lock`.
7. If your own lock exists:
   - inspect related task status in the resolved concrete tasklist file
   - read `tasks/tasklog/<agent>_<taskid>.md` if it already exists
   - if the task is already `done` or `cancelled`, delete the lock
   - otherwise resume that task directly
8. If no active own lock exists, find the first matching `todo` task for your role.
9. Check dependencies.
10. Create lock.
11. Claim task.
12. Initialize `tasks/tasklog/<agent>_<taskid>.md` if it does not exist yet.
13. Commit the tasklist status change from `todo` to `doing`.
14. Execute.
15. Run tests.
16. Self-review.
17. If refactor is needed, repeat execute -> test -> self-review until no refactor is needed.
18. After done a task, the executor must commit all changed files with message describing what was done. Default taskexec commit rules:
   - tasklist file commit: include `tasklist filename + overall task title/description`
   - task commit: include `<agent>_<taskid>` like `codex_t03160314.p010`
   - if the task row is a same-workflow follow-up bugfix, keep `FIX:<source_taskid>` in the task row `Description` or `Detail`; do not require an `ISSUE-ID` in the task commit message
19. record the real task commit hash in `tasks/tasklog/<agent>_<taskid>.md`
20. update tasklist ,if had commit you must update hash to the tasklist 
21. Remove lock.

## Locked Task Rule

- Lock is never taken over by another AI.
- Each AI scans only its own lock first.
- One AI can have only one active lock at a time.
- If an AI already has an unfinished lock task, it must continue that task first.
- Lock filename format looks like `<agent>_<taskid>.lock`.
  - Example: `codex_t03160314.p010.lock`
  - Example: `claude_t03160314.p011.lock`

## Claim Rule

When claiming a task, update only these tasklist fields:

- `Owner`
- `Status`
- `Claim`
- `Finish`
- `Report`
- `Git`
- `Review`
- `Score`

Claim format:

`startat:YYMMDDHHMMSS <agent> tasks/tasklog/<agent>_<taskid>.md`

## Execution Loop

If the task is already claimed by the current AI, do not restart the full claiming flow. Go directly to:

1. execute
2. verify
3. self-review
4. repeat if refactor is needed
5. Commit all changed files with a message describing what was done. Commit rules:
   - tasklist file modification commit: `tasklist filename + overall task title/description`
   - task commit: include `<agent>_<taskid>` like `codex_t03160314.p010`
   - if it is a same-workflow follow-up bugfix task, the task row `Description` or `Detail` must already carry `FIX:<source_taskid>`
6. update tasklist (include commit hash)
7. remove lock

## Testing Rule

A task cannot become `done` unless required verification passes.

Examples:

- `pytest`
- `npm test`
- `go test ./...`
- project-specific manual validation with evidence

If no tests exist, add minimal validation or document manual verification evidence.

## Self Review Rule

Required checks:

- requirement satisfied
- verification passed
- no obvious duplicate logic left unhandled
- naming consistent
- no required refactor remains
- task log is complete

Review outcomes:

- `pass:no-refactor-needed`
- `pass:minor-refactor-done`
- `partial:needs-followup`

## Definition Of Done

A task is `done` only when all are true:

- implementation complete
- required tests passed
- self review says no further necessary refactor
- task log written and updated with the real task commit hash
- task commit is created
- tasklist row is updated
- tasklist state commit is created
- lock is removed

## Task Status Set

Allowed task statuses:

- `todo`
- `doing`
- `blocked`
- `partial`
- `pending`
- `done`
- `cancelled`

## Dependency Rule

Before claiming a `todo` task, check `Depends`. If any dependency is not `done`, skip the task.

## Memory Read Rule

Memory root is `memorys/`.

Use:

- `memorys/global.md` for global rules and project memory links
- `memorys/<project-name>/memory.md` for project-specific memory

Do not read unrelated project memory unless explicitly required.
